# basic_demo_sample — Canonical LogicBank Usage Example

**What it is:** Within GenAI-Logic (ApiLogicServer), the **manager** workspace ships sample projects under `prototypes/manager/samples/`. `basic_demo_sample` is the reference sample — it's the source for the `basic_demo` tutorial project (see `add-cust` mechanism, AI-guided tour, in the ApiLogicServer dev-architecture doc) and is the most-tested, most-stable example of "real" LogicBank usage inside a generated project.

**Location:** `org_git/ApiLogicServer-src/api_logic_server_cli/prototypes/manager/samples/basic_demo_sample/`

**Why it matters for LogicBank dev:** This is how LogicBank actually gets wired up in production-shaped code — not just the `examples/` test fixtures in *this* repo. Useful to see real consumer-side expectations (model shape, activation lifecycle, error handling) when changing the engine.

&nbsp;

## Data Model (`database/models.py`)

Plain SQLAlchemy declarative classes, generated by introspection/GenAI then hand-editable. Five tables: `Customer`, `Order`, `Item`, `Product`, `SysEmail`.

Key conventions observed:
- `Base = SAFRSBaseX` (normally) or `TestBase` (when `APILOGICPROJECT_NO_FLASK` is set) — lets the same model file run rules both inside Flask and in plain test-data-loading scripts, outside any web context.
- Derived/aggregate columns are **materialized as real columns**, not computed properties — e.g. `Customer.balance`, `Order.amount_total`. This is mandatory: LogicBank rules (`Rule.sum`, `Rule.formula`) populate physical columns; there's no virtual-attribute mechanism. (Matches the GenAI prompt-engineering rule noted in ApiLogicServer's dev-architecture.md: "if you create sum/count/formula rules, you must create a corresponding column.")
- Relationships use `Mapped[List[...]]` + `back_populates` (SQLAlchemy 2.0 style) — both directions declared explicitly (parent `OrderList` on `Customer`, child `customer` on `Order`).
- Foreign keys are plain `Column(ForeignKey(...))`, not the relationship object — rules reference columns/relationships, not raw FKs.

&nbsp;

## The Rules (`logic/logic_discovery/place_order/check_credit.py`)

This is the "5 lines replace 200" canonical example referenced throughout GenAI-Logic CE and marketing:

```python
def declare_logic():
    Rule.constraint(validate=models.Customer,
        as_condition=lambda row: row.balance <= row.credit_limit,
        error_msg="Customer balance exceeds credit limit")
    Rule.sum(derive=models.Customer.balance, as_sum_of=models.Order.amount_total,
        where=lambda row: row.date_shipped is None)
    Rule.sum(derive=models.Order.amount_total, as_sum_of=models.Item.amount)
    Rule.formula(derive=models.Item.amount, as_expression=lambda row: row.quantity * row.unit_price)
    Rule.copy(derive=models.Item.unit_price, from_parent=models.Product.unit_price)
```

Maps directly onto the Logic-Walkthrough wiki's forward-chaining example (OrderDetail/Item qty change → Amount → Order total → Customer balance) — this sample *is* that example, in its actual shipped form.

&nbsp;

## Rule Discovery (`logic/logic_discovery/auto_discovery.py`)

GenAI-Logic doesn't require all rules in one `declare_logic.py`. Convention: **one file per use case**, named for it (e.g. `check_credit.py`), placed anywhere under `logic/logic_discovery/` (subdirectories OK, e.g. `place_order/check_credit.py`).

`discover_logic()`:
1. Walks `logic/logic_discovery/` recursively
2. For every `.py` file (except `auto_discovery.py`, `__init__.py`), dynamically imports it and calls its `declare_logic()` function
3. Separately, if a `wg_rules/` folder exists (WebGenAI-exported rules), discovers/loads those too — either from a single `active_rules_export.py` (local/exported run) or by walking the folder (inside WG itself, calling `init_rule()` per file)

This discovery step runs from the top-level `logic/declare_logic.py` (`from logic.logic_discovery.auto_discovery import discover_logic; discover_logic()`), which is itself the function passed to `LogicBank.activate()` as the `activator=`.

Top-level `logic/declare_logic.py` is also where **cross-cutting Python customization** lives (not expressible as declarative rules) — e.g. this sample's `send_order_to_shipping` (`Rule.after_flush_row_event`, sends a Kafka message when an order ships) and `handle_all` (`Rule.early_row_event_all_classes` — generic stamping/optimistic-locking/grant-checking applied to every class). This is the "logic == rules + Python" extensibility model.

## Activation (`config/activate_logicbank.py`)

The actual `LogicBank.activate()` call — the bridge between "rules are declared" and "the engine is listening":

```python
LogicBank.activate(session=session,
                   activator=declare_logic.declare_logic,
                   constraint_event=constraint_handler,
                   aggregate_defaults=aggregate_defaults,
                   all_defaults=all_defaults)
```

Notable production-hardening around the bare `LogicBank.activate()` call that LogicBank's own README/wiki doesn't need to show:
- `APILOGICPROJECT_DISABLE_RULES` env var — short-circuits activation entirely (rules off)
- `AGGREGATE_DEFAULTS` / `ALL_DEFAULTS` env vars — toggle LogicBank's default-rule-inference behavior
- Catches `LBActivateException` specifically — surfaces `invalid_rules` and `missing_attributes` separately, with a link to recovery docs; this is GenAI-Logic's hook for catching malformed GenAI-generated rules (mismatched attribute names, etc.) and feeding them back into a fixup loop (`als genai-utils --fixup`)
- Suppresses the raise when `VERIFY_RULES=True` or running inside WebGenAI (`WG_PROJECT=True`) — lets WG's rule-verification pass collect errors without crashing the process

## The `genai-utils --fixup` Loop (concrete example)

`build_and_test/genai-logic/README.md` walks through a real failure/recovery case worth keeping verbatim — it's the clearest illustration of `LBActivateException` surfacing through to an end user:

```
Logic Bank Activation Error -- see https://apilogicserver.github.io/Docs/WebGenAI-CLI/#recovery-options
Invalid Rules:  [AttributeError("type object 'Customer' has no attribute 'balance'")]
Missing Attrs (try genai-logic genai-utils --fixup): ['Customer.balance: constraint']
```

This happens when a rule references an attribute the data model doesn't have (typically: GenAI declared a rule like `Rule.constraint` against `Customer.balance` but never added the `balance` column to `models.py`/test data). `genai-utils --fixup` collects model + rules + test data, asks ChatGPT to resolve the missing columns, and produces a `docs/fixup/response_fixup.json` used to rebuild the project. Confirms `LBActivateException.invalid_rules` / `.missing_attributes` (documented in `basic_demo_sample.md`'s Activation section above) are exactly what's surfaced here — this is a **consumed, user-visible error format**, not just an internal exception shape.

&nbsp;

## Why This Matters for LogicBank Engine Changes

- `LBActivateException.invalid_rules` / `.missing_attributes` are a **consumed public API** — don't rename/restructure without checking this call site (and the WG fixup flow built on it)
- The `activator=` callback pattern (a function, not a static rule list) is how GenAI-Logic defers rule loading until `activate()` is called — relevant if changing activation timing/ordering
- `aggregate_defaults` / `all_defaults` flags are consumer-facing knobs into LogicBank's default-rule inference — check `rule_bank_setup.py` / activation code before changing their semantics

## Relation to `examples/nw` (in this repo)

`basic_demo_sample` (5 tables: Customer, Order, Item, Product, SysEmail) is a **deliberately smaller** version of this repo's own `examples/nw` (Northwind: Category, Customer, CustomerDemographic, Department, EmployeeAudit, Product, Region, Shipper, Supplier, Territory, CustomerCustomerDemo, Employee, OrderClass, EmployeeTerritory, Order, OrderDetail — 16+ tables). Same check-credit pattern at its core (`Customer.Balance` / `CreditLimit`, `Order.AmountTotal`, `OrderDetail.Amount` = `Quantity * UnitPrice`, `OrderDetail.UnitPrice` copied from `Product`), but nw layers on every relationship edge case LogicBank needs to support, while `basic_demo_sample` only exercises the simple/default case.

### How the model files differ (`examples/nw/db/models.py` vs `basic_demo_sample/database/models.py`)

| Aspect | `basic_demo_sample` | `examples/nw` |
|---|---|---|
| **FK → relationship mapping** | Always 1 FK column → 1 relationship, default `back_populates`, no `foreign_keys=` needed | Several relationships need explicit `foreign_keys=` because of ambiguity (see below) |
| **Multiple FKs to same parent table** | None | `Employee.WorksFor` and `Employee.OnLoan` are **two separate FK columns, both → `Department.Id`** — `Department.EmployeeWorksForList` / `EmployeeOnLoanList` each need `foreign_keys='[Employee.WorksFor]'` / `'[Employee.OnLoan]'` to disambiguate which FK the relationship uses |
| **Self-referencing (recursive) relationship** | None | `Employee.Manager` (parent) / `Employee.Manages` (children) — both sides hand-written with explicit `primaryjoin='foreign(Employee.ReportsTo) == Employee.Id'` and `remote_side=`; **not** derivable from a plain `ForeignKey()` column declaration alone |
| **Virtual relationship — no DB-level FK at all** | None | `EmployeeAudit.Employee` ← `EmployeeAudit.EmployeeId` has **no `ForeignKey()` on the column** (`EmployeeId = Column(Integer)`, not `Column(ForeignKey('Employee.Id'))`); relationship exists purely via `primaryjoin='foreign(EmployeeAudit.EmployeeId) == Employee.Id'`. Explicit test comment: `# test manual virtual relationship (no FK in db)` |
| **Renamed/overridden accessors** | Accessor names always match the natural relationship name (`order`, `customer`, `product`) | `Order.OrderHeader` (not `order`) on `OrderDetail`, `Order.SalesRep` (not `employee`) — accessor name deliberately decoupled from both the FK column name and the target class name |
| **Class name ≠ table name** | Always equal | `OrderClass` class ↔ `__tablename__ = 'OrderZ'` — explicit test of name-mapping independent of class/table identity |
| **SQL-expression virtual attributes** | None | `Customer.total_ordered_sql` and `Employee.order_count_sql` are SQLAlchemy `column_property()` (raw correlated-subquery SQL), not LogicBank rules — used to test that LogicBank rules can read/reference attributes that are themselves *not* LogicBank-derived |
| **Bad-rule / missing-attribute test scaffolding** | None | `LOGICBANK_LOAD_BAD_RULES` env var path deliberately declares rules with typo'd attribute names, to test `LBActivateException` (see `readme_dev.md` → `test_missing_attrs` special case) |
| **Cascade declared explicitly** | Not present | `cascade="all, delete"` on `Customer.OrderList` and `Order.OrderDetailList` |

### Why this matters for the GenAI-Logic bug

A reported LogicBank bug from GenAI-Logic may stem from **relationship-modeling assumption mismatches** — i.e., GenAI-Logic's generated models (which look like `basic_demo_sample`'s simple, default-`back_populates`, single-FK-per-parent style) hitting a LogicBank code path that was only ever exercised against nw's more exotic patterns above (or vice versa: nw-only patterns like multi-FK-to-same-parent or virtual/no-FK relationships not being replicated in GenAI-Logic-generated models, so a real-world equivalent goes untested).

**When debugging:** check whether the failing case involves any of the rows in the table above — especially multiple FKs to the same parent table, self-referencing relationships, or `foreign_keys=`/`primaryjoin=` overrides — since `basic_demo_sample` (and likely most GenAI-Logic-generated projects) never exercises those paths.
